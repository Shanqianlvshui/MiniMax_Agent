# Demo: STM32F103C8T6 USB CDC Driver

## Objective

Use the multi-agent workflow system to develop and validate a USB CDC ACM virtual serial driver for an STM32F103C8T6 minimum system board.

The demo must verify the full workflow:

- Official-source research.
- Evidence and assumption tracking.
- Controlled CubeMX `.ioc` modification.
- Official STM32CubeMX code generation.
- User-code implementation.
- Build and flash.
- Real board USB CDC behavior validation.
- Reviewer pass/fail/needs_human.
- Final evidence-backed report.

## Locked Workflow

```text
Editor: VS Code
Extension: STM32CubeIDE for Visual Studio Code extension pack
Project mode: STM32CubeMX project
Build: ST official VS Code/CMake/CLT workflow
Toolchain: STM32CubeCLT / arm-none-eabi-gcc
Debug/Flash: ST-LINK + STM32CubeProgrammer CLI where available
Firmware package: STM32CubeF1
USB class: CDC ACM virtual COM port
```

## Target Hardware

```text
MCU: STM32F103C8T6
Board: STM32F103C8T6 minimum system board
USB: Device FS via PA11/PA12
Debugger: ST-LINK over SWD
```

## Required Hardware Confirmations

These are blocking if unknown:

```text
Exact MCU marking/model
Board revision or board link/schematic if available
HSE crystal frequency
USB D- wired to PA11
USB D+ wired to PA12
USB D+ pull-up behavior
3.3 V power path
USB GND and board GND common
ST-LINK/SWD availability
BOOT0/BOOT1 boot mode
LED pin and active level if LED commands are enabled
```

If a detail is missing and affects implementation or validation, Researcher creates a blocker and the task enters human input.

## Official Evidence Requirements

Researcher must gather official evidence for:

- STM32F103C8T6 USB device capability.
- USB FS pin mapping for PA11/PA12.
- Clock requirement for USB 48 MHz.
- RCC/PLL configuration constraints for the selected HSE.
- STM32CubeF1 USB Device CDC middleware usage.
- ST official VS Code/CubeMX/CMake workflow commands.
- STM32CubeMX command-line generation support.

Allowed high-authority sources:

- ST datasheet for STM32F103x8/xB.
- ST reference manual RM0008.
- ST errata for STM32F103x8/xB if relevant.
- ST UM1718 STM32CubeMX user manual.
- ST STM32CubeF1 package documentation and examples.
- ST official VS Code extension documentation.
- ST official community or knowledge base as secondary official evidence.

## CubeMX Configuration Scope

V1 only allows a structured STM32F103C8T6 USB CDC intent:

```json
{
  "target": "stm32f103c8t6_usb_cdc",
  "hse_hz": 8000000,
  "usb_class": "CDC_ACM",
  "toolchain": "CMake"
}
```

The actual HSE value must come from user confirmation or board evidence.

Allowed `.ioc` changes:

```text
RCC/HSE selection
PLL source and multiplier
SYSCLK target
USB clock target
APB1/APB2 prescalers
PA11 USB_DM
PA12 USB_DP
USB peripheral/device mode
USB_DEVICE middleware
CDC class
USB interrupt if required by CubeMX/F1 generation
Project toolchain/settings required by the official VS Code flow
```

Not allowed in v1:

```text
Arbitrary .ioc key/value edits
Other USB classes
RTOS
DMA
Custom USB descriptors beyond the minimum needed
Bootloader changes
Option bytes
Linker/memory layout changes
Unconfirmed pin remaps
```

After `.ioc` modification, Executor must run STM32CubeMX CLI generation. If the toolchain cannot run headlessly on the current machine, the task enters `needs_human` and asks the user to run generation manually, then resumes from the generated files.

## Code Modification Rules

Executor may modify:

- `Core/Src/*` and `Core/Inc/*` only in safe user-owned regions or new files.
- `USB_DEVICE/App/*` user code sections.
- `USB_DEVICE/Target/*` only when evidence and generated boundaries permit.
- Project-level docs/artifacts generated by the task.

Executor must not:

- Edit CubeMX generated regions outside `USER CODE BEGIN/END`.
- Delete generated middleware files.
- Hand-edit `.ioc` through generic patch.
- Change startup/linker files unless a future task explicitly supports it.

## Firmware Behavior Protocol

The demo firmware implements a simple text protocol over USB CDC:

```text
PING\n
 -> PONG\n

INFO\n
 -> STM32F103C8T6 USB CDC <firmware_version>\n

ECHO <payload>\n
 -> <payload>\n
```

If LED pin is confirmed:

```text
LED ON\n
 -> OK\n

LED OFF\n
 -> OK\n
```

If LED pin is not confirmed, LED commands are not part of the acceptance criteria.

Implementation guidance:

- CDC receive callback should buffer data and set flags.
- Main loop should parse commands and transmit responses.
- Handle both `\n` and `\r\n`.
- Avoid RTOS in v1.
- Keep buffers small and explicit.

## Hardware Validation

Success is based on real hardware behavior, not just code generation.

Validation events:

```text
build_passed
flash_passed
usb_enumerated
com_port_detected
serial_ping_passed
serial_info_passed
serial_echo_passed
led_test_passed
```

V1 supports semi-automatic validation:

- Build is run through the official VS Code/CMake/CLT workflow.
- Flash is run through STM32CubeProgrammer CLI or an ST-LINK-compatible official path.
- USB enumeration is checked through Windows device queries when possible.
- COM port testing uses a serial tool such as pyserial when possible.
- If automatic detection fails, GUI asks the user to provide logs, screenshots, COM port, or manual pass/fail.

Reviewer must distinguish:

```text
firmware_build_passed
firmware_flashed
usb_enumerated
serial_protocol_passed
board_test_not_run
```

If real-board validation has not run, final status cannot claim hardware success. The maximum status is `firmware_ready_for_board_test`.

## Reviewer Checks

Reviewer must check:

- Task matches USB CDC ACM goal.
- Critical hardware facts have official evidence.
- HSE and clock configuration are confirmed.
- `.ioc` edits are in the v1 whitelist.
- CubeMX generation completed or was manually confirmed.
- User code stays inside safe regions.
- Build result is recorded.
- Flash result is recorded when board is connected.
- USB enumeration result is recorded.
- Serial protocol test result is recorded.
- Unconfirmed assumptions do not affect pass status.
